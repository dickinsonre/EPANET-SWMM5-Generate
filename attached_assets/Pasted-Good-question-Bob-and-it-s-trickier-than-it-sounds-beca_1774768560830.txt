Good question, Bob — and it’s trickier than it sounds because EPANET will happily “run” a model that produces nonsense. A model that returns zero errors isn’t the same as a model that’s physically meaningful. Here’s how I’d think about it from first principles:
Topological validity comes first. Every network needs at least one fixed-grade node (reservoir or tank) — without it, EPANET has no hydraulic reference point and the solver has nothing to anchor pressures to. Every node must be reachable from at least one source through open links. Disconnected nodes are the #1 silent killer — EPANET will report “system unbalanced” but won’t always tell you why. Run a graph traversal from each source and confirm every node is touched.
Check that demands actually pull water. A model with zero demands everywhere will converge beautifully and tell you nothing. Verify that at least some junction nodes have nonzero base demands, and if you’re using demand patterns, confirm the pattern multipliers aren’t all zero for the timestep you care about. A common trap: the default pattern is all 1.0s, which is fine, but if you assign a named pattern that doesn’t exist in [PATTERNS], EPANET silently uses zero demand.
Hydraulic convergence isn’t enough — check the physics. After a successful run, validate these:
	∙	Pressures should be positive at demand nodes (negative pressure means the network can’t deliver water there, which may be physically correct but usually signals a modeling error in a test model)
	∙	Velocities should fall in a reasonable range — below about 0.5 ft/s you get stagnation and sedimentation concerns; above 10 ft/s something is likely wrong with pipe sizing or elevation data
	∙	Head loss across any single pipe shouldn’t exceed the total available head in the system
	∙	Tank levels should stay between their min and max over an extended-period simulation — if a tank drains to minimum and stays there, you’ve got a supply deficit
The continuity check is your ground truth. EPANET doesn’t report a continuity error percentage the way SWMM does, but you can verify mass balance manually: total flow into the system from reservoirs should equal total demand plus any flow into tanks (for EPS at any timestep). If those don’t balance, something is disconnected or a valve/pump status is blocking flow.
Pump and valve sanity checks. If the model has pumps, verify the pump curve covers the operating point — EPANET will extrapolate beyond the curve and give you garbage. Check that the pump’s head at zero flow exceeds the static head it needs to overcome. For check valves and PRVs/PSVs, confirm the settings make physical sense (a PRV set to 200 psi in a 40 psi system does nothing; one set to 5 psi in a 40 psi system might choke the network).
The practical test battery I’d run:
	1.	Steady-state first — does it converge in under 40 trials? If it needs more, something is ill-conditioned
	2.	Extended period (24–72 hrs) — do tank levels cycle reasonably, or do they crash/overflow?
	3.	Zero-demand test — set all demands to zero; pressures everywhere should equal the hydraulic grade of the source minus elevation. If they don’t, you have closed pipes or bad check valves
	4.	Fire-flow test — spike demand at one node; does the system respond plausibly?
	5.	Isolation test — close one key pipe; does the model correctly show which nodes lose service?
The zero-demand test is the most underrated diagnostic. It strips away all the complexity and tells you whether the pipe connectivity and elevations are self-consistent. If static pressures don’t match what you’d calculate by hand (source HGL minus node elevation), you’ve found your problem before anything else muddies the water.​​​​​​​​​​​​​​​​
